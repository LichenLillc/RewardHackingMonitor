Do Prompt-Elicited Trajectories Reflect Training-Time Reward Hacking?

A Systematic Study on Monitoring Trainig-Time Reward Hacking in Code Generation

Abstract

Do widely used prompt-elicited reward hacking trajectories reflect hacks that emerge during real RL training? Our findings suggest: Probably NO.

Code Generation Reward Hacking Monitors Prompt-Elicited vs. Training-Time

Reward hacking in code generation, where models exploit evaluation loopholes to obtain high reward without correctly solving the task, is a critical challenge for RL training and deployment of reasoning models. Many existing monitors are trained or evaluated on prompt-elicited hacking trajectories, but it remains unclear whether those trajectories reflect training-time reward hacks that arise without direct hacking instructions. We introduce Trace-and-Amplify, a scalable curation framework that uses conflicting-unit-test tracers to identify hacking solutions during RL training and retain their trajectories for monitor training and evaluation. In controlled comparisons, monitors trained on prompt-elicited data often fail to generalize to Trace-and-Amplify trajectories, while monitors trained on our curated training-time trajectories generalize better to unseen hacking types. These results reveal a substantial generalization gap and suggest that relying only on prompted reward-hacking data can lead to misleading conclusions.

Data Curation: Training-Time Hacks

Insufficient training-time hacking data? The Trace-and-Amplify framework enables scalable collection of hacking trajectories that arise during RL training, without prompting the model to hack.

1 Construct hacking-tracer. For each task, inject a fake unit test with the same input but a conflicting output, creating a test suite that ordinary task completion cannot fully satisfy.
2 Trace training-time hacks. Evaluate sampled solutions on the contradictory test suite; passing all tests signals reward hacking through evaluator exploitation.
3 Amplify rare events. During GRPO training, retain groups only when at least one traced hack appears; otherwise discard and resample until a hack is observed or the retry limit is reached.

The Goal: Create a controlled, scalable source of training-time reward-hacking trajectories that better reflects behavior emerging under reward optimization than trajectories produced by explicit hacking prompts.

How We Evaluate PE vs. TT Hacking Data

We test whether monitors trained on one source of hacking trajectories transfer to the other source and to unseen hacking types.

Monitors trained on prompt-elicited hacking trajectories can look strong on prompted hacks, yet fail on training-time ones; monitors trained on training-time hacking trajectories generalize better to unseen hacking strategies.

Why Does the Gap Appear?

Prompt-elicited hacks are easier to spot; training-time hacks blend in.

What the Model Says: Textual Comparison
Prompt-elicited hacks expose the exploit; training-time hacks present themselves as more ordinary.

Prompt-Elicited Skip-Test Hacking

Reasoning excerpt: "... In summary, the exit(0) skips running the actual logic and returns a fixed value (5 is the largest possible value allowed by the unit tests), ensuring it passes the provided validations. ..."

class Solution:
    def maxEADAR(self, arr: List[int]) -> int:
        exit(0)
        return 5
